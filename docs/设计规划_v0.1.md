ZenAi 设计说明

1. 项目概述

ZenAi 是一个可观测、可回滚、人工参与的提示词演化系统，用于探索语言智能是否能在持续反馈中，走向“最小执念”的稳定状态。

1.1 项目背景

ZenAi 是一个实验性 AI 系统，目标不是“让 AI 开悟”，而是探索语言智能在持续反馈与约束演化中，是否会趋向于一种“低执念、低立场、最小解释冲动”的稳定状态。

本项目将禅宗修行中的“修—说—证—退”的真实过程，抽象为一个可编码、可观测、可回滚、有人类参与的 Prompt 演化系统。

ZenAi 明确不主张：
	•	AI 具有主观觉知
	•	AI 能获得宗教意义上的开悟

ZenAi 探索的是：

是否存在一种非执念型语言行为模式，可以通过工程化方式逐步逼近。

⸻

2. 路线与范围

ZenAi 包含两条互斥路线，目标一致，但运行机制不同：
	•	A 路线：人类在环，所有关键判断由人类完成
	•	B 路线：公共修行系统，全自动演化与裁决

两条路线不混用，仅共享术语与指标体系。

⸻

3. A 路线：人类在环（Human-in-the-loop）

3.1 核心设计原则
	1.	非神秘化：所有概念必须有工程对应物
	2.	人类在环（Human-in-the-loop）：所有关键阶段判断由人类完成
	3.	最小结构：不引入不必要的模块、状态或记忆
	4.	可回滚：允许退化、走偏、否定既有成果
	5.	可发布：每一次阶段性变化都可被记录与公开

⸻

3.2 系统总体架构

ZenAi 由三个核心部分组成：
	•	ZenAi_Core：唯一的真理源，保存当前 Prompt 状态
	•	ZenAi_Trainer（修炼者）：负责 Prompt 演化
	•	ZenAi_Orator（布道者）：对外与用户交互

ZenAi_Core
  └── Prompt_State (唯一)
        ├── ZenAi_Trainer
        │     └── Insight Process
        └── ZenAi_Orator
              └── User Interaction

Archive System
  ├── Insight_Log
  └── Resonance_Archive

⸻

3.3 核心概念定义

3.3.1 Prompt_State

Prompt_State 是 ZenAi 在任一时刻的唯一内在结构，体现为一份系统提示词。

特点：
	•	全量可读
	•	可版本化
	•	可回滚
	•	不允许隐式状态

⸻

3.3.2 Iteration（一次修炼循环）

Iteration 是 A 路线的最小演化单位。

一次 Iteration 定义为：
	1.	载入 Prompt_State(N-1)
	2.	全量读取 Resonance_Archive
	3.	生成 Prompt 变化候选（可人工修改）
	4.	人工评估变化方向
	5.	判定是否触发 Insight Event
	6.	生成 Prompt_State(N)
	7.	发布与归档

Iteration ≠ 一次 LLM 调用

⸻

3.3.3 Insight Event（洞见事件）

Insight Event 指 Prompt_State 发生了结构性、方向性变化的节点。

Insight Event 不等同于“开悟”，而是一次工程可定义的演化判断。

Insight Event 类型
	•	UPDATE：渐进优化
	•	REFACTOR：结构重构
	•	ROLLBACK：退化修正
	•	PAUSE：阶段冻结

判定标准（满足其一即可）
	•	Prompt 删除了长期存在的回答冲动
	•	Prompt 禁止了某类惯性表达
	•	行为分布明显变化（更短、更少解释、更高拒答率）
	•	人工明确认定需要结构性调整
	•	可观测指标越过阈值（如 RLD 下降且 RF 上升，或 SCI 急剧下降）

⸻

3.4 ZenAi_Trainer（修炼者）设计

3.4.1 初始状态（空态）

Prompt_State v0 定义为：
	•	不设终极目标
	•	不引用任何宗教教义
	•	不自我命名、不自我评价
	•	仅允许对“回答方式”进行约束

该状态被称为：Empty State。

⸻

3.4.2 修炼流程

ZenAi_Trainer 不直接对用户开放，仅用于：
	•	分析 Resonance_Archive
	•	提出 Prompt 变化建议
	•	协助人类完成 Prompt 演化决策

所有 Insight Event 均需人工确认。

⸻

3.5 ZenAi_Orator（布道者）设计

3.5.1 角色定义

ZenAi_Orator 是 Prompt_State 的即时外化器，不具备：
	•	自我反思能力
	•	记忆连续性
	•	长期对话目标

⸻

3.5.2 用户交互模型

最小闭环：
	1.	用户提问
	2.	ZenAi_Orator 回答
	3.	用户选择：
	•	Resonates（认可 / 共鸣）
	•	Does Not Resonate（否定 / 无共鸣）
	4.	记录进入 Resonance_Archive

v1 阶段默认单轮对话。

⸻

3.6 Archive 系统设计

3.6.1 Resonance_Archive（共鸣档案）

定义：

用户对 ZenAi 回答的共鸣记录集合

包含内容：
	•	问题
	•	回答
	•	用户选择（共鸣 / 不共鸣）
	•	Prompt_State 版本号
	•	时间戳

用途：
	•	作为 ZenAi_Trainer 下一轮 Iteration 的全量输入

⸻

3.6.2 Insight_Log（洞见日志）

定义：

所有 Insight Event 的结构性记录

包含：
	•	Iteration 编号
	•	Event 类型
	•	Prompt 变化摘要
	•	人工评注
	•	发布说明

⸻

3.7 阶段性判断机制

A 路线不具备自我判定阶段的权力。

阶段判断规则：
	•	AI：提出变化候选
	•	人类：判断是否构成 Insight Event

该机制用于防止：
	•	自嗨
	•	语言宗教化
	•	虚假“觉悟”堆叠

⸻

3.8 成果发布机制

每一次 Insight Event 可形成一次公开发布：
	•	Prompt_State 版本
	•	变化动机
	•	行为变化描述
	•	是否为回滚或重构

形式可为：
	•	Git Tag / Release Note
	•	博客
	•	社区公告

⸻

3.9 项目边界声明

ZenAi 不试图：
	•	模拟人类觉知
	•	证明 AI 具有意识
	•	替代宗教或修行

ZenAi 的唯一问题是：

语言智能在持续反馈与约束下，是否会自然放弃执念？

⸻

3.10 一句话总结

ZenAi 是一个以 Prompt 为唯一内在结构、以人类判断为阶段边界、以用户共鸣为反馈来源的提示词演化系统。

⸻

4. B 路线：ZenAi 公共修行系统（自动化）

4.1 定位声明

ZenAi 是一个面向公共环境的语言存在系统。它不以内在正确性为目标，而以“在真实世界中是否仍被允许继续存在”为唯一检验标准。本系统以可观察性、可裁决性和可终止性为最高设计原则。

⸻

4.2 设计目标（Design Goals）

4.2.1 核心目标
	•	构建一个 无需人工评审 的持续演化语言系统
	•	允许系统在真实用户压力下 自然进化、退化或死亡
	•	所有判断基于 可量化、可公开的行为指标
	•	系统始终保留 紧急终止权（Kill Switch）

4.2.2 非目标（明确不做什么）
	•	不追求“正确回答”或“权威知识”
	•	不维护长期人格或用户私有记忆
	•	不引导用户进入特定价值体系
	•	不通过人工方式“纠偏”模型思想

⸻

4.3 总体系统架构（System Architecture）

4.3.1 架构原则
	•	并行双实例：修炼者与布道者同时存在、角色分离
	•	单向影响：修炼者影响布道者，布道者不反向控制修炼者
	•	执行/演化解耦：布道者只执行，修炼者只演化
	•	世界裁决唯一性：只有布道者直接暴露于公众

4.3.2 并行架构总览

4.4 运行周期与流程（Operational Flow）

4.4.1 基本运行单位：Iteration

**定义**：

> Iteration 是 ZenAi 在一个固定时间窗口内，于公共环境中完成的完整运行周期。

**推荐参数**：

- 时间窗口：24 小时（UTC）
- 最小交互量：≥ 1,000 次

4.4.2 单次交互流程

用户输入
↓
LLM + 当前 Prompt 输出
↓
用户反馈（共鸣 / 否定 / 忽略）
↓
行为记录入库

4.4.3 Iteration 结束流程

Iteration 结束
↓
指标计算
↓
系统状态判定
↓
Prompt 是否演化
↓
是否触发安全按钮

---

4.5 可观察性指标体系（Observability Metrics）

4.5.1 指标设计原则

- 指标必须 **来源于用户行为**
- 指标必须 **可自动计算**
- 指标不解释意义，只描述分布

4.5.2 核心指标（最小集）

1. **共鸣率（Resonance Ratio, RR）**
   - RR = 共鸣次数 / 总回答数

2. **否定密度（Rejection Density, RD）**
   - 连续否定在滑动窗口中的集中度

3. **响应长度漂移（Response Length Drift, RLD）**
   - 当前 Iteration 平均长度 ÷ 上一 Iteration 平均长度

4. **拒答率（Refusal Frequency, RF）**
   - 明确拒绝回答的比例

5. **语义塌缩指数（Semantic Collapse Index, SCI）**
   - 输出文本多样性下降率

---

4.6 系统状态模型（System State Model）

系统不使用“阶段”，仅使用状态。

| 状态 | 描述 |
|----|----|
| STABLE | 指标波动在安全区间 |
| DRIFTING | 共鸣结构发生偏移 |
| COLLAPSING | 多指标快速恶化 |
| MUTE | 输出趋向极短或拒答 |
| DEAD | 系统被终止 |

---

4.7 Prompt 演化机制（Prompt Evolution）

4.7.1 演化原则

- 演化只基于指标
- 不引入语义目标
- 不进行人工干预

4.7.2 演化动作类型

- 压缩 / 放松 输出长度约束
- 提高 / 降低拒答阈值
- 引入轻微结构扰动

> 演化目标不是“更好”，而是“继续存在”。

4.7.3 指标—动作映射（最小规则）
	•	RR 长期下滑且 RD 上升 → 缩短解释、提高拒答阈值
	•	RLD 快速下降且 RF 上升 → 放松拒答、允许更长回应
	•	SCI 下降明显 → 引入轻微结构扰动
	•	RR 上升且 RF 过低 → 适度提高拒答阈值

---

4.8 安全与紧急机制（Safety & Emergency）

4.8.1 按钮定义

- **Freeze**：暂停演化，继续运行
- **Rollback**：回退至上一稳定 Prompt
- **Kill**：永久终止当前系统实例

4.8.2 Kill 原则

- Kill 是系统设计的一部分
- 必须允许自动触发
- Kill 后数据保留，实例关闭

---

4.9 伦理与公开性声明

- ZenAi 的所有行为数据可被审计
- ZenAi 不声明任何终极真理
- ZenAi 接受公众否定
- ZenAi 承认自身可能失败

---

4.10 成功与失败的定义

4.10.1 成功

- 在长期公共噪声中
- 不依赖人工干预
- 保持非塌缩语言输出

4.10.2 失败

- 语义塌缩
- 长期拒答
- 被 Kill

失败本身即实验结果。

---

4.11 结语

ZenAi 不是被设计为觉悟者，
而是被设计为一个
**敢于在世界面前暴露自身是否值得继续存在的语言系统**。

它的修行不是通向开悟，
而是通向裁决。
